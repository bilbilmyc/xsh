# XSH 安全策略

## 支持范围

当前只保证 `main` 分支上的最新代码得到处理。XSH 仍处于个人使用和持续打磨阶段，不承诺稳定版本的长期安全支持。

## 报告安全问题

请不要在公开 issue 中粘贴密码、私钥、凭据数据库、真实主机信息或可利用细节。优先通过 GitHub Security Advisories 的私密报告入口提交；如果仓库尚未启用该入口，请先联系仓库维护者，再提供最小化的复现信息。

报告应包含：

- 受影响的提交或版本；
- 操作系统和架构；
- 不包含秘密的复现步骤；
- 影响范围和建议修复方向。

## 安全边界

- SSH 密码和 Key Passphrase 只由 XSH 自建的本地加密凭据库管理，不写入系统钥匙串、导出 JSON、工作区恢复数据或普通日志。
- 普通会话导出不包含秘密；凭据备份使用独立密码加密的备份格式。
- Host Key 首次出现需要用户确认，Host Key 发生变化时默认阻断连接。
- 快捷命令栏和命令中心拒绝保存高置信度明文凭据、Token、私钥和 `sshpass -p` 等内容。
- 不要把真实凭据用于自动化测试；测试应使用内存存储、文档地址和占位符。

## 已知限制

个人版 macOS 构建使用 ad-hoc 签名，不提供 Apple 公证；Windows 构建默认未进行代码签名。不要把未经签名的构建当作面向不受信任用户的正式发布包。
